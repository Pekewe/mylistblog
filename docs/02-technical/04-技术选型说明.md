## 2. 选型理由

### 2.1 Node.js + Express

**为什么不用其他语言？**
- 前后端统一使用 JavaScript，降低学习成本
- Express 是最成熟、最轻量的 Node.js Web 框架
- 个人站点并发低，Node.js 异步模型完全够用

### 2.2 SQLite（sql.js）

**为什么不用 MySQL / PostgreSQL？**

| 对比项 | SQLite | MySQL |
|--------|--------|-------|
| 安装配置 | 零配置，无需安装 | 需要安装服务 |
| 部署复杂度 | 一个文件即可 | 需要单独配置 |
| 备份 | 拷贝一个文件 | 需要 dump 或工具 |
| 并发能力 | 低（个人站点够用） | 高 |
| 适用场景 | 个人站点、小型应用 | 企业级应用 |

**为什么选 sql.js 而不是 better-sqlite3？**
- sql.js 是纯 JS 实现，**不需要 C++ 编译**，安装即用
- better-sqlite3 需要 node-gyp 编译，在部分系统上可能报错
- 个人站点数据量小，sql.js 的性能差距可忽略

### 2.3 bcryptjs

**为什么不用原生 bcrypt？**
- bcryptjs 是纯 JS 实现，无编译依赖
- 原生 bcrypt 需要 C++ 编译工具链
- 10 轮哈希对个人站点完全够用

### 2.4 Vanilla JS（纯原生 JavaScript）

**为什么不用 React / Vue？**

| 对比项 | Vanilla JS | React / Vue |
|--------|-----------|-------------|
| 文件大小 | 0KB（无需引入库） | 几十 KB |
| 学习成本 | 零（只需懂 JS） | 需要学习框架概念 |
| SEO | 天然支持 | 需要 SSR |
| 开发效率 | 低（手动操作 DOM） | 高（声明式 UI） |
| 适用场景 | 小型页面 | 复杂交互应用 |

本项目的页面交互简单（列表展示 + 表单提交），Vanilla JS 完全够用，且维护成本最低。

## 3. 弃用的技术

| 技术 | 原因 | 替代方案 |
|------|------|----------|
| `data.js`（localStorage 数据） | localStorage 在浏览器端不安全且不可共享 | 替换为 `api.js` + 后端 API |
| jQuery | 现代浏览器原生 API 已足够 | 原生 `fetch` / `XMLHttpRequest` |
